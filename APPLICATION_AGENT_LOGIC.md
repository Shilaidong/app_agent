# Terra-Edu 申请 Agent 软件逻辑说明

更新时间：2026-06-08

本文不是代码说明，也不是底层架构文档。它只描述这个软件“想做什么、用户怎么用、Agent 怎么思考、每一步应该如何推进、遇到问题怎么处理”。它的目标是让另一个开发者、产品设计师，或者另一个 Agent 能读懂 Terra-Edu 申请 Agent 的完整行为语言，并据此重构、复刻或继续改造这个产品。

如果要一句话概括：Terra-Edu 申请 Agent 是一个给留学申请顾问使用的桌面申请助理。顾问输入学生、材料文件夹、学校、项目和申请链接后，软件把后续资料整理、学生档案生成、缺失项检查、申请要求研究、申请平台填表、保存进度和总结输出都交给 OpenCode Agent 执行。

## 1. 产品定位

这个软件不是一个普通聊天机器人，也不是一个只会打开网页的自动化脚本。它更像一个“本地申请工作台 + 申请专用 Agent”。

它服务的对象是留学申请顾问。顾问的真实工作场景通常是：

- 手里有一个学生的材料文件夹。
- 要帮这个学生申请一个学校和一个项目。
- 需要先读材料、整理信息、判断缺什么。
- 再去学校申请系统里开账号、填表、上传材料、保存草稿。
- 遇到不确定信息时要问学生或家长。
- 最终提交、付款、推荐信邀请这些高风险动作必须人工处理。

Terra-Edu 申请 Agent 要做的是把这个流程变成一个可重复、可追踪、可继续的 Agent 工作流。顾问不需要一步步指挥“先读文件，再分类，再生成表”，而是在创建任务后让 Agent 自动开始。

## 2. 核心原则

### 2.1 顾问只负责目标和确认

顾问提供任务边界：谁、申请哪里、申请什么项目、申请链接在哪里、材料文件夹在哪里。任务启动后，Agent 应主动推进，不要求顾问再输入第一条指令。

顾问需要参与的地方主要是登录申请平台、回答学生材料里没有的信息、确认多个可能选项、处理验证码或 MFA、以及执行最终提交、付款、不可逆推荐信邀请等人工动作。

### 2.2 原始材料永远只读

学生原始材料文件夹不能被 Agent 直接修改。任务开始时，软件会把原始文件复制到申请工作区的备份目录。之后所有整理、分类、生成、修改、补充材料处理，都只发生在申请工作区里。

这样做的原因很简单：学生原始材料是证据源，不能被自动化流程污染。

### 2.3 先结构化，再填表

Agent 不能一上来就打开网页盲填。正确顺序是先整理学生材料，形成学生申请档案，研究学校和项目要求，明确缺失信息和缺失材料，然后再进入申请平台填表。

网页填表只是整个申请流程的一部分，不是全部。

### 2.4 不确定就问，不编造

如果材料中没有某个字段，Agent 不能为了完成表格而瞎填。比如父母信息、推荐人邮箱、考试日期、工资、签证状态、具体地址、项目轮次、是否豁免语言成绩等，只要缺证据，就必须记录为缺失项或向顾问确认。

### 2.5 可以保存草稿，不能做不可逆动作

Agent 允许做读取材料、整理材料、生成文档、打开申请系统、填写可确认字段、上传可确认材料、保存草稿、记录失败原因和截图留证。

Agent 不允许自动做最终提交申请、付款、发送不可逆推荐信邀请、保存明文账号密码，或在无证据时填写不确定信息。

### 2.6 所有动作都要留下状态

Agent 每完成一步，都要让任务状态、进度文件、日志、审计记录能够反映出来。顾问看到的进度不应该只是聊天窗口里的文字，而应该和本地工作区的状态保持一致。

## 3. 用户看到的软件

### 3.1 第一屏

软件打开后默认进入任务入口，而不是通用 OpenCode 项目页。

第一屏只做两件事：新建申请任务，或者读取已有申请任务。这里不应该做成营销首页，也不应该让文科顾问看到一堆开发者概念。

### 3.2 新建任务

新建任务时，固定要求顾问填写学生姓名、学生资料文件夹、申请学校、申请项目或专业、申请类型、申请平台链接。

可以选填截止日期、备注、登录方式、申请平台账号、是否记住申请平台密码、输出语言、是否允许上传材料、本次任务目标。

这些字段不是为了做表单好看，而是为了给 Agent 一个足够稳定的任务边界。

### 3.3 读取已有任务

软件不应该每次打开都默认恢复上次任务。更合适的方式是：打开后仍然在入口页，顾问可以选择“新建”或“读取已有申请”。

已有申请列表按学生姓名分组。点击某个学生后，再看到这个学生下面的学校和项目。这样比把所有历史申请平铺在一起更符合顾问的工作习惯。

空任务、没有学校的假任务、没有真正启动的残留任务，不应该每次自动出现在列表里。

### 3.4 任务工作台

进入任务后，顾问看到的是一个申请工作台，核心区域是 OpenCode Agent 对话。

工作台应该包括当前任务信息、申请列表、缺失统计、快捷操作、进度条或进度时间线、AI 对话工作区、生成结果文件、日志和截图入口、安全边界提示。

AI 对话工作区应该占主要空间，因为这个软件的核心不是传统后台表单，而是 Agent 真实执行申请任务的过程。

### 3.5 快捷操作

快捷按钮不是单独业务页面。它们只是把标准申请指令发送给 OpenCode Agent。

常见按钮包括整理资料、生成学生档案、检查缺失、生成信息收集表、生成材料收集表、开始填表、继续填表、生成 Word 清单、总结进度。

顾问点按钮，本质上是在用自然语言命令 Agent，而不是让外壳自己偷偷完成一套独立流程。

### 3.6 对话显示

顾问需要看到 Agent 在干什么，但不需要看到所有技术细节。

Agent 的自然语言说明默认展开，重要问题和确认项默认展开，命令行输出、工具返回、JSON、审计细节默认折叠。Markdown 应正常渲染，表格不能窄到每列只能显示几个字，新消息出现时应平滑滚动，顾问点选确认卡片后不应该突然跳到对话第一行。

## 4. 本地申请工作区

每个申请任务都有一个独立工作区。工作区名称一般由学生、学校、项目组合而成。

工作区固定包含这些部分：

- `00_original_backup`：原始材料备份。任务开始时从学生文件夹复制而来，之后不再修改。
- `01_classified_materials`：分类后的材料。
- `02_generated`：Agent 生成给顾问、学生、家长看的文件。
- `03_state`：任务状态、缺失项、申请进度、申请要求、凭据状态、审计记录。
- `04_logs`：Agent 日志和 CUA 日志。
- `05_screenshots`：必要时保存的申请平台截图。
- `06_new_materials`：顾问或学生后续补充的新材料。

这个工作区的意义是让申请过程从“聊天记忆”变成“可追踪的申请档案”。即使对话上下文变长、Agent 发生压缩、软件重启，任务仍然可以继续。

## 5. 任务启动后的默认流程

创建任务后，软件会自动进入 AI 申请工作区，并自动启动默认申请流程。顾问不需要再输入“开始吧”。

### 第一步：读取任务信息

Agent 先确认任务边界：学生是谁、学校是什么、项目是什么、申请链接是什么、原始材料在哪里、是否允许上传、登录方式是什么。

如果关键信息缺失，Agent 应优先提出确认问题，而不是直接开始猜。

### 第二步：建立申请工作区

软件或 Agent 创建标准工作区，并把原始学生文件夹复制到 `00_original_backup`。这一阶段的关键要求是：原始文件夹不被修改。

### 第三步：整理和分类材料

Agent 读取备份材料，根据文件名、内容和上下文分类。

常见分类包括身份材料、学术材料、语言成绩、文书材料、推荐相关材料、财务材料、申请平台相关材料、其他材料、需要人工复核的材料。

不能判断用途的材料不应该硬塞到某一类，而应该进入需要复核。

### 第四步：生成学生申请档案

Agent 从材料里提取学生信息，生成 `student_profile.md`。

这个文件是后续填表的核心参考。它应该包含学生基本信息、身份信息、教育经历、成绩和学位信息、语言或标化考试信息、工作或实习经历、文书和简历信息、可用于申请系统填写的字段、材料来源路径、缺失项、不确定项、申请目标。

如果材料是扫描 PDF 或图片，Agent 应使用可用的 OCR、本地工具、MCP 或浏览器能力提取文字。提取不稳定时，要标记可信度和不确定项。

### 第五步：研究学校和项目要求

Agent 不能只依赖顾问输入的申请链接。它还需要研究学校和项目的官方要求。

优先方式是读取申请链接和项目官网页面。如果申请链接信息不足，再搜索学校项目要求。

研究结果要保存为 `application_requirements.json` 和 `application_requirements.md`。

要求记录应包括来源 URL、抓取时间、要求类型、材料要求、字段要求、截止日期、轮次、语言成绩要求、标化成绩要求、推荐信要求、不确定项和可信度。

如果官网信息和申请系统页面冲突，以申请系统实际页面为优先，但要在日志里说明。

### 第六步：检查缺失内容

Agent 根据学生档案和申请要求，生成缺失项记录。

缺失项分为缺材料、缺信息、需要确认、平台内待查看、可豁免但需顾问确认。

缺失项的核心状态文件是 `missing_items.json`。后续所有缺失清单都应该从这个文件生成，而不是每次靠聊天重新总结。

### 第七步：生成收集表和 Word 清单

Agent 应生成 `info_collection_form.md`、`material_collection_form.md` 和 `missing_materials.docx`。

Word 清单必须面向顾问、学生和家长，避免技术语言。它不应该写“JSON 字段缺失”之类的话，而应该写“请补充当前或最近一段工作经历的职位名称、公司名称、起止时间”。

### 第八步：打开申请平台并开始填表

Agent 使用 CUA 打开申请平台。

如果需要登录，Agent 等待顾问登录或使用顾问本地保存的账号凭据辅助登录。验证码、MFA、邮箱确认等仍然交给顾问。

登录后，Agent 观察页面，识别当前页面、表单、弹窗、必填项、按钮和已有内容。

### 第九步：真人式填写、复查、保存

Agent 不应该像机械脚本一样“点一个字段、输一个值、直接保存”。更可靠的方式是模仿真人：

1. 观察当前页面。
2. 识别本页字段和弹窗。
3. 填 1 到 3 个字段。
4. 立刻复查这些字段是否真的填上了。
5. 观察是否因为选项变化出现了新字段。
6. 继续填下一组字段。
7. 保存前检查必填空项和错误提示。
8. 保存后确认页面确实保存成功。

如果遇到下拉框、学校搜索、日期选择、原生系统菜单、自动补全菜单，Agent 应使用更稳的选择策略，而不是只靠鼠标坐标。

选择策略的优先级应包括直接选择网页中的选项值、设置可访问性控件的值、使用键盘输入搜索词、使用方向键和回车、选择后读取显示值确认、必要时尝试视觉或截图兜底。

如果出现 Chrome 的“离开此网站？”提示，默认应选择取消或留在页面，避免未保存内容丢失。

### 第十步：阶段总结和继续申请

当某一阶段完成或遇到阻塞，Agent 要生成总结。

总结应说明已完成什么、哪些页面已保存、哪些材料已上传、哪些字段仍缺失、哪些动作需要顾问处理、下一步建议是什么。

如果顾问后来把新材料放入 `06_new_materials`，Agent 应重新读取补充材料，更新学生档案、缺失项和申请进度，然后继续填表。

## 6. OpenCode Agent 的角色

这个软件的关键不是外壳自己完成业务逻辑，而是把申请任务交给 OpenCode Agent。

外壳负责创建任务、创建本地工作区、准备 OpenCode 会话、写入申请专用 Prompt、Skills、Commands、Custom Tools、展示 Agent 对话和结果、提供必要的本地工具能力。

OpenCode Agent 负责读材料、调工具、制定计划、更新进度、生成文件、研究官网要求、判断缺失项、驱动 CUA 填表、遇到不确定问题时提问、总结结果。

因此，重构这个软件时，不应把所有业务都写死在外壳里。外壳应该提供工具和边界，真正的申请判断交给 Agent。

## 7. 默认模型

当前默认使用 OpenCode Go 入口下的 DeepSeek V4 Pro。

选择更强模型的目的不是为了让 UI 更漂亮，而是让 Agent 在长任务、材料理解、官网要求判断、动态表单应对和上下文恢复上更稳。

如果没有配置可用模型或 API Key，软件应明确提示顾问配置或登录，而不是静默失败。

## 8. 申请专用 Prompt 的语言

申请 Agent 的 Prompt 应该像一位资深申请顾问和自动化助理的工作规范，而不是像一段程序说明。

它应该明确告诉 Agent：你正在处理一个真实学生的申请任务；顾问已经提供了任务目标，不要等待第一条指令；先建立计划，再推进任务；每一步都要更新本地状态；原始材料不可修改；所有生成文件要写到规定位置；不确定信息必须提问或记录缺失；不允许最终提交、付款、发送不可逆推荐信邀请；可以填写、上传、保存草稿；遇到浏览器表单时，按真人式循环处理；工具失败时先换策略，不要立刻放弃；失败也要记录原因，让任务可以继续。

它还应该要求 Agent 少讲技术细节，多讲顾问能理解的话。

## 9. 12 个 Skills 的行为

申请 Agent 内置的 Skills 不是装饰，而是让 Agent 在不同阶段知道该怎么做。

### 9.1 任务初始化

负责确认任务边界、创建十步计划、写入初始状态、准备后续动作。

### 9.2 学生文件读取

负责读取原始材料备份，识别文件类型，提取可读文本，发现扫描件时尝试 OCR 或可用工具。

### 9.3 工作区构建

负责保证标准目录、状态文件、日志文件和生成目录都存在。

### 9.4 材料整理

负责把材料分类到对应目录，并生成材料索引。不能判断的材料进入需要复核。

### 9.5 学生资料生成

负责生成 `student_profile.md`，把可确认信息、来源路径、缺失项、不确定项都写清楚。

### 9.6 申请目标分析

负责研究学校、项目、截止日期、轮次、申请系统、材料要求和填写要求。

### 9.7 缺失内容记录

负责维护 `missing_items.json`，并区分缺材料、缺信息、需要确认和平台内待查看。

### 9.8 Word 清单生成

负责把缺失项转换成顾问、学生、家长能直接阅读的 Word 清单。

### 9.9 CUA 填表

负责驱动浏览器填写申请平台，按观察、填写、复查、保存的循环推进。

### 9.10 材料上传

负责上传可确认材料，记录成功、失败、原因和页面证据。

### 9.11 补材料后继续

负责读取 `06_new_materials`，更新档案和缺失项，然后继续申请流程。

### 9.12 任务总结

负责生成阶段性或最终总结，说明已完成、未完成、阻塞、风险和下一步。

## 10. 官方 OpenCode 工具的使用方式

这个软件应该充分利用 OpenCode 官方工具，而不是把 Agent 限制成普通聊天。

相关官方文档：

- [OpenCode Tools](https://opencode.ai/docs/zh-cn/tools/)
- [OpenCode Custom Tools](https://opencode.ai/docs/zh-cn/custom-tools/)
- [OpenCode Skills](https://opencode.ai/docs/zh-cn/skills/)
- [OpenCode Commands](https://opencode.ai/docs/zh-cn/commands/)
- [OpenCode Permissions](https://opencode.ai/docs/zh-cn/permissions/)

### 10.1 todowrite

Agent 启动后应先创建十步任务清单。这个清单不是给用户看的装饰，而是 Agent 自己的执行轨道。

每完成关键节点，Agent 应同步更新任务进度。这样顾问看到的软件进度和 Agent 内部计划才一致。

### 10.2 webfetch 和 websearch

申请要求必须优先来自官方页面。

Agent 应先读取申请链接和学校项目页面。如果页面内容不足，再搜索学校项目要求。所有结果都要落到申请要求文件，并保留来源链接。

### 10.3 question

当 Agent 遇到不确定事项，应使用问题卡片让顾问确认。

问题卡片应像顾问确认事项，而不是暴露底层工具格式。它应包含一句清楚的问题、两到三个推荐选项，并允许顾问自定义回复。

适合使用 question 的场景包括是否选择某一轮次、是否豁免语言成绩、申请类型不确定、推荐人信息缺失、多个学校或项目名称相似、表单字段有多个可能答案。

## 11. 申请专用 Custom Tools 的意义

Custom Tools 是这个软件的“申请动作语言”。它们让 Agent 不只会说话，还能用标准方式操作本地任务。

`workspace` 用于创建和检查申请工作区，保证目录结构和状态文件存在。

`materials` 用于复制、分类、索引、补充和复核学生材料。

`documents` 用于生成学生档案、信息收集表、材料收集表、Word 缺失清单和任务总结。

`state` 用于更新任务状态、缺失统计、当前阶段和进度记录。

`requirements` 用于保存学校和项目要求，包括来源、字段需求、材料需求和不确定事项。

`login` 用于处理申请平台账号状态。账号和密码只能保存在本机安全存储或钥匙串，不应出现在聊天、日志或工作区明文文件里。

`cua` 用于浏览器填表、页面观察、字段填写、选项选择、保存验证、弹窗处理、截图和失败记录。

`risk` 用于拦截高风险动作。Agent 即使在网页上看到最终提交、付款或推荐信邀请按钮，也不能自动执行。

所有关键工具动作都应该留下审计记录。审计记录要能回答哪个会话做的、哪个 Agent 做的、在哪个工作区做的、做了什么动作、成功还是失败、失败原因是什么。

审计不是给顾问日常阅读的，但它是排查问题和重构系统时的重要证据。

## 12. CUA 真人式填表逻辑

CUA 是这个软件最关键也最容易出问题的部分。它不能只是一个“点这里、输入那里”的工具，而应该是一套真人式浏览器操作循环。

### 12.1 页面观察

每进入一个页面，Agent 先观察当前页面标题、当前 URL、是否有弹窗、是否有系统提示、是否有必填项、哪些字段已经有值、哪些字段为空、哪些按钮可以保存、是否有上传区域、是否有错误提示。

常规观察应优先使用可访问性树和网页 DOM，不默认截图。这样可以减少 macOS 截图权限弹窗。

### 12.2 小步填写

Agent 每次只填 1 到 3 个字段，然后复查。

这样做比一次性填完整页更稳，因为申请系统经常会根据某个选项动态显示新字段。

例如选择国家后出现州，选择学校后出现专业，选择考试类型后出现日期和分数结构。Agent 填完关键选项后必须重新观察页面。

### 12.3 选择下拉和浮动菜单

申请平台里常见的下拉菜单有普通网页下拉、原生系统菜单、搜索式学校选择、动态自动补全、日期拆分选择、Slate 风格弹窗表单。

Agent 不应只靠鼠标坐标点击。更稳的方法是先输入候选词，再用键盘方向键、回车、可访问性选择或网页值设置来完成，并在选择后读取显示值确认。

如果选择失败，Agent 应换策略，而不是立即告诉顾问“CUA 受限无法填写”。

### 12.4 保存前复查

点击保存前，Agent 必须检查红色必填项是否为空、页面是否显示 “All fields required”、是否有隐藏必填字段、弹窗内是否还有未填项、当前填写内容是否符合字段格式。

如果保存失败，Agent 不应反复点保存，而应读取错误提示，补齐字段或记录缺失项。

### 12.5 保存后验证

只有当页面确认保存成功，或字段出现在已保存列表中，才算保存完成。不能因为 Agent 自己说“我保存了”就把页面记为已保存。

### 12.6 弹窗处理

如果出现浏览器提示或系统弹窗，Agent 应先判断类型。

“离开此网站？”默认取消或留在页面，防止丢失未保存内容。validation alert 要读取错误文案并回到表单修正。权限弹窗要记录是哪个程序请求权限并提示顾问处理。下载、上传、选择文件弹窗要按当前任务判断是否继续。

### 12.7 截图策略

正常填表不应默认截图。截图只在连续失败需要证据、上传材料后需要留证、保存成功后需要留证、顾问明确要求截图、或 Agent 无法从可访问性和网页结构判断页面状态时使用。

如果 macOS 弹截图权限，日志应说明：实际请求权限的可能是 CUA Driver、Chrome Helper 或浏览器自动化组件，不一定是 Terra-Edu App 本身。

## 13. 登录与账号密码

软件有两类登录：顾问登录 Terra-Edu 申请 Agent，以及申请平台账号登录。

顾问登录用于识别用户和控制 AI 使用额度。申请平台登录用于进入学校申请系统。

当前直发版本的顾问登录不要求客户自己配置 Supabase。安装包里已经带有 Supabase 登录和额度所需的公开运行配置，顾问打开软件后可以直接进入登录流程。

申请平台账号可以由顾问输入，并在本机加密保存，以便下次继续。但密码不能明文写入聊天、日志、任务文件或工作区。

验证码、MFA、邮箱确认等属于人工确认环节，Agent 不应试图绕过。

## 14. AI 额度逻辑

顾问登录后可以使用 AI 额度。当前产品语言可以理解为：每个顾问有一组初始额度，用于完成若干份申请任务。

额度不应简单按“问一次算一次”理解，因为一次申请任务里模型会读材料、研究官网、调用工具、生成文件、填表和总结。更合理的方式是按模型实际消耗的 token 折算为额度。

产品层可以把额度表达成更容易理解的数字，例如“剩余额度”或“可继续处理约几份申请”。当额度不足时，软件应提示顾问联系微信 `shilaidong` 获取更多额度。

当前直发版本把 Supabase 登录、额度提示、默认初始额度和额度不足联系入口作为产品内置能力，而不是让客户自己去准备环境变量。

## 15. 直发版本与内置运行配置

当前仓库和安装包采用“小范围客户直发”的交付方式。这个选择的目标是：客户下载、安装、打开后，就能看到登录、能使用 AI、能创建任务，而不是先配置 API key、Supabase 或模型 provider。

因此，当前版本会把运行时私有配置纳入仓库和安装包：

- `packages/desktop/resources/private/opencode-go-key.txt`：默认 OpenCode Go 模型路线所需 key。
- `packages/desktop/resources/private/supabase-public.json`：Supabase 登录、额度和联系入口所需公开运行配置。

这两个文件被打进桌面安装包：

- macOS 包内路径：`Contents/Resources/private/opencode-go-key.txt` 和 `Contents/Resources/private/supabase-public.json`。
- Windows 包内路径：`resources/private/opencode-go-key.txt` 和 `resources/private/supabase-public.json`。

这不是一个适合公开镜像的开源分发形态。它适合你当前的小范围顾问客户交付。未来如果要公开仓库、给更多人 fork，或进入更正式的商业分发，应该先旋转这些 key，或者改成后端中转和授权系统。

换句话说：当前版本优先保证“到手即用”；未来商业版再把密钥、额度、授权和模型调用迁移到更可控的服务端。

## 16. 任务继续与上下文恢复

申请任务很长，不能依赖一次聊天上下文完成。

Agent 继续任务时，应读取当前任务状态、申请进度、缺失项、学生档案、申请要求、审计记录、最近日志，然后判断应该从哪里继续，而不是重新开始。

如果 OpenCode 对话上下文发生压缩，Agent 也应该把本地状态文件当作事实来源。

## 17. 顾问确认卡片

当 Agent 需要顾问判断时，应该用确认卡片，而不是把问题埋在长段文字里。

好的确认卡片应该问题简短明确、选项少而清楚、推荐选项放在前面、允许自定义回复、顾问点完后对话不乱跳、回答后 Agent 自动继续。

这会让文科顾问感觉自己是在审核申请，而不是在调试系统。

## 18. 生成文件说明

`student_profile.md` 是学生申请档案，也是后续填表的核心资料源。

`info_collection_form.md` 是信息收集表，用于向学生或家长追问缺失信息。

`material_collection_form.md` 是材料收集表，用于告诉学生还需要补哪些文件。

`missing_materials.docx` 是 Word 缺失材料清单，应该最适合直接发给学生、家长或顾问团队。

`task_summary.md` 是阶段总结或最终总结，说明任务已经做到哪里、还差什么、下一步是什么。

`task_state.json` 是任务当前状态。

`missing_items.json` 是缺失项事实来源。Word 清单和信息表都应该从这里派生。

`application_progress.json` 是申请进度事实来源。它记录当前页面、已填字段、已验证字段、已保存页面、阻塞弹窗、最后观察时间等。

`application_requirements.json` 和 `application_requirements.md` 是学校和项目要求。前者用于 Agent 判断，后者用于顾问阅读。

`agent_log.md` 和 `cua_log.md` 是 Agent 行为日志和浏览器自动化日志。技术细节默认不打扰顾问，但必须存在。

## 19. 安全边界

这个软件的安全边界必须清晰可见，且不只停留在提示文案。

安全边界至少有三层：Prompt 层禁止、工具层识别并阻止、总结层提示人工处理。

禁止自动执行的动作包括最终提交申请、付款、发送不可逆推荐信邀请、保存明文密码、在无证据时填写不确定字段。

如果网页按钮写着 Submit、Pay、Payment、Finalize、Invite Recommender、Send Request 等，Agent 应默认视为高风险，除非它只是保存草稿或非最终动作，并且已经明确验证。

## 20. UI 体验原则

这个软件的用户大概率不是工程师，所以 UI 应该服务申请流程，而不是展示技术能力。

好的 UI 应该让中间主要区域留给 AI 对话，左侧展示任务和申请列表，右侧展示结果文件，进度区紧凑不抢空间，技术输出默认折叠，顾问确认默认醒目，Markdown 渲染清楚，长回答可以展开，列表按学生组织，可以读取历史任务但不强制恢复上次任务。

## 21. 失败处理

Agent 不能把失败简单甩给顾问。

失败时应先换一种工具方式、重新观察页面、读取错误提示、判断是否是缺信息、判断是否是权限问题、判断是否是高风险动作、记录失败原因、提供下一步建议。

只有在连续失败、需要验证码、需要人工确认、涉及高风险动作或平台确实不支持自动化时，才要求顾问介入。

## 22. 如何让另一个 Agent 模仿这个软件

如果另一个 Agent 想复刻 Terra-Edu 申请 Agent，不需要先复制代码，而应该复制这些行为契约：

1. 任务入口只收集申请必要信息。
2. 每个任务有本地隔离工作区。
3. 原始材料只读备份。
4. 启动后自动创建十步计划。
5. 先整理材料，再生成档案，再研究要求，再填平台。
6. 缺失项必须结构化保存。
7. 顾问确认必须用清楚的问题卡片。
8. 浏览器填表必须观察、填写、复查、保存验证。
9. 高风险动作必须阻止。
10. 任务可中断、可恢复、可总结。

只要这些行为契约成立，底层可以换成别的模型、别的浏览器自动化、别的桌面框架或别的文件存储方式。

## 23. 重构时不要丢掉的语言

重构时最容易丢掉的不是代码，而是产品语言。

这个软件的语言应始终围绕学生、学校、项目、材料、缺失项、申请要求、当前页面、已保存、待确认、需要顾问处理。

尽量少说 JSON、schema、resolver、handler、IPC、DOM exception、accessibility tree、stack trace。

这些技术细节可以进入日志，但不应该成为顾问的主要体验。

## 26. 最后一句话

Terra-Edu 申请 Agent 的本质不是“一个会操作浏览器的软件”，而是一套申请顾问工作语言：把学生材料变成结构化档案，把学校要求变成可执行清单，把申请平台变成可验证的填表流程，把不确定事项交还给顾问确认，把高风险动作留给人。

未来无论怎么重构，只要这套语言不丢，这个产品就还在。
